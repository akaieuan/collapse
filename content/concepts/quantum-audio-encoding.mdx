## The mental shift

Classical audio is a sequence of amplitudes indexed by time. Quantum audio re-expresses that signal as a quantum state — either as probability amplitudes (QPAM/SQPAM) or as bit-quantized sample values stored in an amplitude register (QSM). Encoding is a deterministic builder; the "audio" lives only inside the circuit until you measure (decode) it.

The headline trade-off:

- **Compactness** — QPAM uses just `⌈log₂(N)⌉` qubits for `N` samples. QSM adds an amplitude register of `bit-depth` qubits, lossless but heavier.
- **Read-back** — QPAM/SQPAM are statistical (more shots → smoother reconstruction). QSM is exact in the noiseless limit.
- **Manipulation** — gates applied between encode and decode are how you "process" the audio in the quantum domain.

```python {1-2#imports 4#scheme-load 6#encode 8#aer 10-11#run 13#decode} title="encode_qpam.py"
import numpy as np
import quantumaudio

scheme = quantumaudio.load_scheme("qpam")

circuit = scheme.encode(my_audio_buffer)

from qiskit_aer import AerSimulator
sim = AerSimulator()
job = sim.run(circuit, shots=4096)
counts = job.result().get_counts()

samples = quantumaudio.decode(circuit, shots=4096)
```

<Note id="imports" tip="quantumaudio is the moth-quantum package; numpy is an implicit peer dep">
The package ships its own scheme loaders and a `decode` helper. You bring your own simulator (AerSimulator is the default).
</Note>

<Note id="scheme-load" tip='load_scheme accepts "qpam" | "sqpam" | "msqpam" | "qsm" | "mqsm"'>
The returned scheme object exposes `.encode(audio)` and is what you pass back to `quantumaudio.decode` — the package figures out the matching reconstruction.
</Note>

<Note id="encode" tip="encode() returns a fresh QuantumCircuit each call">
No mutation of the audio buffer. The circuit's qubit count is `ceil(log2(num_samples))` for time, plus the scheme-specific amplitude register.
</Note>

<Note id="aer" tip="AerSimulator is the local Qiskit simulator from qiskit-aer">
Collapse always uses AerSimulator. For real hardware, swap in an IBM Quantum or IQM backend; the rest of this code stays the same.
</Note>

<Note id="run" tip="Shots determine how many times the circuit is sampled">
For QPAM/SQPAM, more shots = smoother decoded waveform. 4096 is a reasonable default for short clips.
</Note>

<Note id="decode" tip="quantumaudio.decode reconstructs samples from the circuit's measurement statistics">
The scheme that built the circuit determines how counts get turned back into samples. Always pair encode and decode with the same scheme.
</Note>

```tsx {1#use-server 4-12#api-route 14-23#client} title="app/api/encode/route.ts + use"
// app/api/encode/route.ts
import { NextResponse } from "next/server";
import { spawn } from "node:child_process";

export async function POST(req: Request) {
  const body = await req.json();
  return new Promise<Response>((resolve) => {
    const child = spawn(".venv/bin/python", ["scripts/encode.py"]);
    let out = "";
    child.stdout.on("data", (c) => (out += c));
    child.on("close", () => resolve(NextResponse.json(JSON.parse(out))));
    child.stdin.write(JSON.stringify(body));
    child.stdin.end();
  });
}

// app/components/EncodeButton.tsx (client)
"use client";
export function EncodeButton({ samples }: { samples: number[] }) {
  async function go() {
    const res = await fetch("/api/encode", {
      method: "POST",
      body: JSON.stringify({ scheme: "qpam", audio: { samples, sampleRate: 8000, channels: 1 } }),
    });
    console.log(await res.json());
  }
  return <button onClick={go}>Encode</button>;
}
```

<Note id="use-server" tip="Route Handlers run on Node — perfect for shelling out to Python">
Collapse's `/api/run` route uses exactly this shape — `runtime: 'nodejs'`, `dynamic: 'force-dynamic'`, Zod-validated body, child_process to a Python sidecar.
</Note>

<Note id="api-route" tip="spawn returns immediately; resolve in the close handler">
Wrap in a Promise so Next can return a real Response. For longer runs add a timeout via setTimeout + child.kill('SIGKILL').
</Note>

<Note id="client" tip='POST a typed body — the server route validates with Zod'>
Always use `'use client'` for interactive components that fetch from API routes — server components can't hold form state.
</Note>

```vue {3-7#composable 9-13#run 16-22#template} title="useQuantumAudio.ts + Encoder.vue"
<script setup lang="ts">
import { ref } from 'vue'

const useQuantumAudio = () => {
  const result = ref<unknown>(null)
  const pending = ref(false)
  return { result, pending }
}

async function encode(samples: number[]) {
  pending.value = true
  const r = await fetch('/api/encode', { method: 'POST', body: JSON.stringify({ scheme: 'qpam', samples }) })
  result.value = await r.json()
  pending.value = false
}
</script>

<template>
  <button @click="encode(samples)" :disabled="pending">
    {{ pending ? 'Encoding…' : 'Encode' }}
  </button>
  <pre v-if="result">{{ result }}</pre>
</template>
```

<Note id="composable" tip="Composables are Vue's hook equivalent — no rules-of-hooks">
For Collapse-style behavior in Vue, extract `useQuantumAudio` into `composables/useQuantumAudio.ts` (Nuxt) or a regular module (plain Vue). Returns refs that the template auto-unwraps.
</Note>

<Note id="run" tip="No deps array, no useCallback — refs track themselves">
The `pending` ref change automatically re-renders the template. Compare to React where you'd need `useState` + setter calls in the right order.
</Note>

<Note id="template" tip="v-if and template auto-unwrap of refs is the readability win">
`{{ result }}` reads `result.value` for you. No `.value` in templates, always `.value` in script.
</Note>

</LangTab>

```vue {2-6#use-fetch 9-12#refresh 16-19#template} title="pages/encode.vue"
<script setup lang="ts">
const { data, pending, error, refresh } = await useFetch('/api/encode', {
  method: 'POST',
  body: { scheme: 'qpam', audio: getCurrentAudio() },
  key: 'encode-current',
})

// Re-run when scheme changes
const scheme = useState('scheme', () => 'qpam')
watch(scheme, () => refresh())
</script>

<template>
  <div>
    <select v-model="scheme"><option value="qpam">QPAM</option><option value="qsm">QSM</option></select>
    <pre v-if="pending">encoding…</pre>
    <pre v-else>{{ data?.circuit }}</pre>
  </div>
</template>
```

<Note id="use-fetch" tip="useFetch handles SSR, hydration, dedup, refs all in one call">
Top-level await works because Nuxt server-renders this. The result is fully serialized and rehydrated on the client. The `key` is the dedup hash.
</Note>

<Note id="refresh" tip="Use refresh() when the URL is fixed but you want to manually rerun">
For URL-reactive refetches (passing a function as the URL), useFetch tracks deps automatically. For manual triggers (button click, scheme change), call refresh().
</Note>

<Note id="template" tip="useState is Nuxt's SSR-safe ref factory — not React's useState">
Nuxt auto-imports `useState`, `ref`, `useFetch`, `useRoute`, etc. The auto-import is ergonomic but reduces grep-ability — get used to checking the auto-import index in @nuxt/devtools.
</Note>
