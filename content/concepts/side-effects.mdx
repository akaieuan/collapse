## What "side effect" means in each world

In React, a side effect runs after the function output is committed to the DOM. In Vue, it runs whenever a tracked dependency changes. In Nuxt, the same Vue model applies, plus first-class data fetching that handles SSR. In Qiskit, "side effect" doesn't really apply at the framework level — circuits run as a single job.

```tsx {1#use-client 5-9#useEffect 11-14#cleanup 17-22#cancellation} title="UserProfile.tsx"
'use client';

import { useEffect, useState } from 'react';

export function UserProfile({ id }: { id: string }) {
  const [user, setUser] = useState<{ name: string } | null>(null);
  useEffect(() => {
    fetch(`/api/users/${id}`).then(r => r.json()).then(setUser);
  }, [id]);

  useEffect(() => {
    const t = setInterval(() => console.log('tick'), 1000);
    return () => clearInterval(t);
  }, []);

  // safer pattern with cancellation:
  useEffect(() => {
    const controller = new AbortController();
    fetch(`/api/users/${id}`, { signal: controller.signal })
      .then(r => r.json())
      .then(setUser)
      .catch(() => {});
    return () => controller.abort();
  }, [id]);

  return <div>{user?.name ?? 'Loading…'}</div>;
}
```

<Note id="use-client" tip="Effects only run in client components">
Server components can't have effects — there's no client runtime to schedule them. If you need to load data on the server, do it directly inside the async server component instead.
</Note>

<Note id="useEffect" tip="The deps array is what makes useEffect tricky">
The effect runs after every render where any value in `[id]` changed. Forgetting to include something is the classic stale-closure bug. The exhaustive-deps lint catches most cases.
</Note>

<Note id="cleanup" tip="Return a function to clean up — runs before next effect or unmount">
React uses the return value as a cleanup. Critical for subscriptions, timers, and listeners. Forgetting it leaks.
</Note>

<Note id="cancellation" tip="Cancellation is your responsibility">
React doesn't cancel in-flight fetches when `id` changes mid-request. Use AbortController in the cleanup, or you risk the older fetch resolving last and clobbering newer state.
</Note>

```vue {3-7#watch 9-15#watchEffect 17-23#async-pattern} title="UserProfile.vue"
<script setup lang="ts">
import { ref, watch, watchEffect } from 'vue'

const props = defineProps<{ id: string }>()
const user = ref<{ name: string } | null>(null)

watch(() => props.id, (newId) => {
  fetch(`/api/users/${newId}`).then(r => r.json()).then(u => user.value = u)
}, { immediate: true })

// Or, dependency-tracking variant:
watchEffect(async () => {
  // automatically re-runs whenever props.id changes
  const r = await fetch(`/api/users/${props.id}`)
  user.value = await r.json()
})

// With cancellation:
watchEffect(async (onCleanup) => {
  const controller = new AbortController()
  onCleanup(() => controller.abort())
  const r = await fetch(`/api/users/${props.id}`, { signal: controller.signal })
  user.value = await r.json()
})
</script>

<template>
  <div>{{ user?.name ?? 'Loading…' }}</div>
</template>
```

<Note id="watch" tip="watch takes a source + a callback">
Explicit dependency. The callback runs when the source changes. `{ immediate: true }` makes it also run on setup, like a useEffect with deps.
</Note>

<Note id="watchEffect" tip="watchEffect tracks deps automatically">
No deps array. Vue records every reactive read inside the function on first run, then re-runs whenever any tracked dep changes. The closest equivalent to React's useEffect-with-perfect-deps, but free.
</Note>

<Note id="async-pattern" tip="onCleanup is passed in — call it to register cleanup">
Vue's cleanup pattern mirrors React's, but the cleanup function is passed as an argument rather than returned. This composes more naturally with async functions.
</Note>

```vue {2-6#useFetch 8-13#refresh-on-change 15-19#async-data} title="pages/users/[id].vue"
<script setup lang="ts">
const route = useRoute()
const { data: user, pending, error, refresh } = await useFetch(
  () => `/api/users/${route.params.id}`,
  { key: 'user' }
)

// Refresh on a custom condition:
watch(() => route.params.id, () => {
  refresh()
})

// Server-only fetch with caching:
const { data: stats } = await useAsyncData('stats', () =>
  $fetch('/api/stats'),
  { server: true, lazy: false }
)
</script>

<template>
  <div v-if="pending">Loading…</div>
  <div v-else>{{ user?.name }}</div>
</template>
```

<Note id="useFetch" tip="useFetch handles SSR, hydration, dedup, and reactivity in one call">
The killer Nuxt primitive. Runs on the server during SSR, serializes the result for the client, and deduplicates the request via the `key`. Returns refs that you can `await` at the top level of `<script setup>` thanks to top-level await.
</Note>

<Note id="refresh-on-change" tip="Pass a function as the URL to make it reactive">
When you pass a `() => string` instead of a string, useFetch tracks the function as a dependency and refetches when its output changes. No manual watch needed — the watch above is for cases where you want extra control.
</Note>

<Note id="async-data" tip="useAsyncData is the lower-level primitive">
Use when you need to fetch from somewhere other than an HTTP endpoint, or compose multiple data sources into one cache entry. `useFetch` is sugar over `useAsyncData` with `$fetch`.
</Note>

<LangTab lang="qiskit">

```python {1-3#imports 5-10#circuit 12-15#submit 17-22#callback-style} title="async_job.py"
from qiskit import QuantumCircuit, transpile
from qiskit_aer import AerSimulator
from qiskit.providers import JobStatus

qc = QuantumCircuit(2, 2)
qc.h(0)
qc.cx(0, 1)
qc.measure([0, 1], [0, 1])

sim = AerSimulator()
job = sim.run(transpile(qc, sim), shots=1024)

print(job.status())
result = job.result()
print(result.get_counts())

# On real hardware, jobs are async:
def on_done(j):
    print("complete:", j.result().get_counts())

job.add_done_callback(on_done)
```

<Note id="imports" tip="No reactive primitives needed — execution is batch-mode">
There's no `useEffect` analog because there's nothing changing while you wait. You build a circuit, submit it, and either block on `result()` or register a callback.
</Note>

<Note id="circuit" tip="Build before submit — no live state">
This is the key difference from JS frameworks: the circuit is constructed in full before any execution. There's nothing to "react to" during construction.
</Note>

<Note id="submit" tip="run() returns a job handle">
The simulator returns a Job object that wraps the asynchronous execution. `.status()` polls; `.result()` blocks until done. This is closer to a Promise than a reactive ref.
</Note>

<Note id="callback-style" tip="add_done_callback is the only 'side effect' pattern">
For real quantum hardware where jobs queue for minutes, you register a callback. The closest analog to a useEffect cleanup would be cancelling the job — `job.cancel()`.
</Note>
