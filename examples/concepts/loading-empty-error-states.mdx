## The mental shift

A data-driven UI has **four** possible visual states, not one:

1. **Loading** — the request is in flight. Show a skeleton, not a spinner if you can avoid it. Skeletons preserve layout; spinners create CLS when content lands.
2. **Empty** — the request succeeded, but the result is `[]` or `{}`. This is not an error. Render a "nothing here yet" with a clear next action.
3. **Error** — the request failed. Render the error in-place with a retry, not a toast that vanishes. Errors that disappear are errors the user can't act on.
4. **Success** — the actual content. The reward state. Don't crowd it with the other three.

Shipping only state 4 (the success path) is the single most common UI bug in research-data UIs. The skill is: every data-fetching component is a 4-branch render, never a 1-branch render.

The pattern shape:

```
const { data, pending, error } = useFetch(...)
if (pending) → <Skeleton />
if (error) → <ErrorBlock onRetry={...} />
if (!data || isEmpty(data)) → <EmptyState />
return <RealContent data={data} />
```

The branching is invariant; the fetch primitive (`useState + fetch`, `useFetch`, `$fetch`) is framework-specific.

```tsx {1-3#types 5-13#fetch-hook 15-22#component 24-44#states} title="components/SearchResults.tsx"
"use client";

import { useEffect, useState } from "react";

type Counts = Record<string, number>;
type FetchState =
  | { kind: "pending" }
  | { kind: "empty" }
  | { kind: "error"; message: string }
  | { kind: "ok"; data: { results: Counts; total: number } };

function useSearch(query: string): { state: FetchState; retry: () => void } {
  const [state, setState] = useState<FetchState>({ kind: "pending" });
  const [nonce, setNonce] = useState(0);

  useEffect(() => {
    let cancelled = false;
    setState({ kind: "pending" });
    fetch(`/api/search/${query}`)
      .then(async (res) => {
        if (cancelled) return;
        if (!res.ok) {
          const detail = await res.text().catch(() => res.statusText);
          setState({ kind: "error", message: detail });
          return;
        }
        const data = await res.json();
        const empty = !data.results || Object.keys(data.results).length === 0;
        setState(empty ? { kind: "empty" } : { kind: "ok", data });
      })
      .catch((err) => !cancelled && setState({ kind: "error", message: String(err) }));
    return () => { cancelled = true; };
  }, [query, nonce]);

  return { state, retry: () => setNonce((n) => n + 1) };
}

export function SearchResults({ query }: { query: string }) {
  const { state, retry } = useSearch(query);

  if (state.kind === "pending") {
    return (
      <div className="h-40 animate-pulse rounded-md bg-muted/40" aria-busy="true" aria-label="Loading results" />
    );
  }
  if (state.kind === "error") {
    return (
      <div className="rounded-md border border-destructive/40 bg-destructive/5 p-4 text-sm" role="alert">
        <p className="font-medium">Search failed</p>
        <p className="mt-1 text-xs text-muted-foreground">{state.message}</p>
        <button onClick={retry} className="mt-2 rounded bg-foreground px-2.5 py-1 text-xs text-background">Retry</button>
      </div>
    );
  }
  if (state.kind === "empty") {
    return (
      <div className="rounded-md border border-dashed p-6 text-center text-sm text-muted-foreground">
        No results yet. Run a search to populate this view.
      </div>
    );
  }
  return <ResultsList results={state.data.results} total={state.data.total} />;
}
```

<Note id="types" kind="core" tip="Discriminated union for state — never four booleans (loading + empty + error + ok)" remember="`{ kind: 'pending' | 'empty' | 'error' | 'ok' }` — one source of truth. Booleans get out of sync.">
The single biggest improvement to this pattern: don't model state as four booleans (`isLoading`, `isEmpty`, `isError`, `data`). Use a discriminated union. The TypeScript compiler enforces exhaustiveness — if you add a fifth state, every consumer of the union needs to handle it. With booleans, you can ship a component that renders nothing because all four are false.
</Note>

<Note id="fetch-hook" kind="core" tip="Extract the state machine into a hook — components stay declarative" remember="useSearch returns { state, retry }. Component reads state.kind. Hook owns the fetch lifecycle.">
The component is dumb: it pattern-matches on `state.kind`. The hook owns cancellation (cancelled flag), retry (nonce trick), and the success/empty/error classification. This separation is what makes the 4-state pattern maintainable — without it, every component reimplements the lifecycle and they drift.
</Note>

<Note id="component" kind="mnemonic" tip="The four branches render in fixed order — pending, error, empty, ok — for cognitive load" remember="if pending → if error → if empty → ok. Same order every component. Reads like a checklist.">
Establishing a convention (pending first, error second, empty third, success last) lets anyone reading the codebase scan a component in two seconds. It also matches the visual hierarchy: pending is "wait," error interrupts, empty is "do something," success is the reward.
</Note>

<Note id="states" kind="gotcha" tip="aria-busy on the skeleton + role='alert' on the error — accessibility is part of the pattern" remember="aria-busy=true on skeleton, role='alert' on error block. Screen readers need both.">
A loading state without `aria-busy` is invisible to screen readers — the user gets confusing silence. An error state without `role="alert"` doesn't get announced. The four-state pattern is also an accessibility pattern: every state needs its ARIA. Skip this and you've optimized for sighted users only.
</Note>

```vue {1-3#imports 5-14#composable 16-23#refresh 26-49#template} title="composables/useSearch.ts + SearchResults.vue"
<script setup lang="ts">
import { ref, watchEffect } from 'vue'

type Counts = Record<string, number>
type State =
  | { kind: 'pending' }
  | { kind: 'empty' }
  | { kind: 'error'; message: string }
  | { kind: 'ok'; data: { results: Counts; total: number } }

const props = defineProps<{ query: string }>()
const state = ref<State>({ kind: 'pending' })
const nonce = ref(0)
function retry() { nonce.value++ }

watchEffect(async (onCleanup) => {
  let cancelled = false
  onCleanup(() => { cancelled = true })
  state.value = { kind: 'pending' }
  try {
    const res = await fetch(`/api/search/${props.query}?n=${nonce.value}`)
    if (cancelled) return
    if (!res.ok) { state.value = { kind: 'error', message: await res.text() }; return }
    const data = await res.json()
    const empty = !data.results || Object.keys(data.results).length === 0
    state.value = empty ? { kind: 'empty' } : { kind: 'ok', data }
  } catch (err) {
    if (!cancelled) state.value = { kind: 'error', message: String(err) }
  }
})
</script>

<template>
  <div v-if="state.kind === 'pending'" class="skeleton" aria-busy="true" aria-label="Loading results" />
  <div v-else-if="state.kind === 'error'" class="error-block" role="alert">
    <p class="font-medium">Search failed</p>
    <p class="muted">{{ state.message }}</p>
    <button @click="retry">Retry</button>
  </div>
  <div v-else-if="state.kind === 'empty'" class="empty-block">
    No results yet. Run a search to populate this view.
  </div>
  <ResultsList v-else :results="state.data.results" :total="state.data.total" />
</template>

<style scoped>
.skeleton { height: 10rem; background: color-mix(in oklab, var(--muted) 40%, transparent); border-radius: var(--radius); animation: pulse 1.5s infinite; }
.error-block { background: color-mix(in oklab, var(--destructive) 5%, transparent); border: 1px solid color-mix(in oklab, var(--destructive) 40%, transparent); padding: 1rem; border-radius: var(--radius); }
.empty-block { border: 1px dashed var(--border); padding: 1.5rem; text-align: center; color: var(--muted-foreground); border-radius: var(--radius); }
@keyframes pulse { 0%, 100% { opacity: 1 } 50% { opacity: 0.5 } }
</style>
```

<Note id="imports" kind="cross" tip="watchEffect with onCleanup is Vue's useEffect + cleanup-fn — semantically identical" remember="watchEffect((onCleanup) => { onCleanup(() => ...) }) — same cancellation pattern as React's useEffect return.">
Vue's `watchEffect` auto-tracks reactive deps (no deps array needed). The `onCleanup` callback runs before the next execution OR on unmount — same role as React's `useEffect` return function. The semantics are identical; only the API surface differs.
</Note>

<Note id="composable" kind="core" tip="Same discriminated union — proof the pattern is framework-agnostic" remember="State type identical to Next. Only the reactive primitive (ref vs useState) and the effect primitive (watchEffect vs useEffect) change.">
This is what cross-stack means: the data shape (the discriminated union) is portable between frameworks. You can move this type definition between files written in React, Vue, or even Svelte and the type would still be correct. Frameworks differ in primitives; the *state model* doesn't have to.
</Note>

<Note id="refresh" kind="mnemonic" tip="nonce-bump pattern for retry — works in any framework with reactive deps" remember="nonce: ref → increment to refetch. watchEffect re-runs because it reads nonce.value. Same trick as React's setNonce.">
The "nonce bump" pattern: a ref that increments to trigger a re-fetch. The watchEffect reads `nonce.value` so it re-runs when nonce changes. Same pattern in React (a state variable in the useEffect deps array). Either way, retries are explicit and traceable — no cache invalidation needed.
</Note>

<Note id="template" kind="gotcha" tip="v-if / v-else-if chain — order matters, exits at first match" remember="v-if pending → v-else-if error → v-else-if empty → v-else ok. Same order as the React if-chain. Discipline.">
Vue's `v-if/v-else-if` chain compiles to the same if-else-if as the React conditional return. If you reorder them (error before pending, etc.), you get subtle bugs — an error during loading might briefly render. Stick with the convention: pending → error → empty → success.
</Note>

```vue {1-13#use-fetch 16-43#template} title="components/SearchResults.vue"
<script setup lang="ts">
type Counts = Record<string, number>
const props = defineProps<{ query: string }>()

const { data, pending, error, refresh } = await useFetch<{ results: Counts; total: number }>(
  () => `/api/search/${props.query}`,
  { server: false, immediate: true }
)

const isEmpty = computed(() => !data.value?.results || Object.keys(data.value.results).length === 0)
</script>

<template>
  <div v-if="pending" class="skeleton" aria-busy="true" aria-label="Loading results" />
  <div v-else-if="error" class="error-block" role="alert">
    <p class="font-medium">Search failed</p>
    <p class="muted">{{ error.statusMessage ?? error.message }}</p>
    <button @click="refresh()">Retry</button>
  </div>
  <div v-else-if="isEmpty" class="empty-block">
    No results yet. Run a search to populate this view.
  </div>
  <ResultsList v-else :results="data!.results" :total="data!.total" />
</template>

<style scoped>
.skeleton { height: 10rem; background: color-mix(in oklab, var(--muted) 40%, transparent); border-radius: var(--radius); animation: pulse 1.5s infinite; }
.error-block { background: color-mix(in oklab, var(--destructive) 5%, transparent); border: 1px solid color-mix(in oklab, var(--destructive) 40%, transparent); padding: 1rem; border-radius: var(--radius); }
.empty-block { border: 1px dashed var(--border); padding: 1.5rem; text-align: center; color: var(--muted-foreground); border-radius: var(--radius); }
@keyframes pulse { 0%, 100% { opacity: 1 } 50% { opacity: 0.5 } }
</style>
```

<Note id="use-fetch" kind="cross" tip="useFetch returns { data, pending, error, refresh } — Nuxt's all-in-one is the four-state pattern built in" remember="useFetch destructures into the four states (sort of). data is the value; isEmpty is derived. Refresh = retry.">
This is where Nuxt earns its reputation: `useFetch` returns refs for `pending`, `error`, and `data` directly. You don't write a discriminated union; you destructure. The trade-off: the empty case isn't built-in — you have to derive it with `computed`. Acceptable, because empty is application-logic, not fetch-state.
</Note>

<Note id="template" kind="core" tip="Same four-branch v-if chain — Nuxt doesn't change the rendering pattern, just the data primitive" remember="Template body identical to Vue tab. Only the script changes — useFetch replaces watchEffect. Pattern invariant.">
This is the cross-stack invariant in action: the template that handles the four states is *byte-for-byte* the same as the Vue tab. The Nuxt-specific change is one composable swap in the script. Everything visible to the user is identical. That's what good cross-framework portability looks like.
</Note>
