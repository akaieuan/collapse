## The mental shift

Many processes return a distribution, not a single value. A 1000-sample A/B test doesn't return "A" or "B" — it returns `{"A": 503, "B": 497}`. The honest design move is to render the distribution as it is: bars whose heights are proportional to sample counts, with the raw numbers visible.

Three rules for designing this primitive:

1. **Don't fake precision.** If the underlying data is a distribution, don't collapse it to a single average. Render the distribution.
2. **Don't hide the sample count.** A 100-sample distribution and a 100,000-sample distribution look identical if you only normalize. Show the absolute counts somewhere — they're the confidence signal.
3. **Pick a stable bar order.** Iteration order of dict keys is not stable across runs in some implementations. Sort by label (lexicographic) or by count (descending) — pick one and commit.

The data structure is universal: `Record<string, number>` (or `Dict[str, int]` in Python). Every visualization library and every hand-rolled SVG renderer reads the same shape. So the skill is "given this shape, here's how each stack idiomatically renders it."

```tsx {1-3#types 5-15#component 17-34#svg 36-40#legend} title="components/DistributionChart.tsx"
type Counts = Record<string, number>;
type Props = { counts: Counts; samples: number; height?: number };

export function DistributionChart({ counts, samples, height = 160 }: Props) {
  const entries = Object.entries(counts).sort(([a], [b]) => a.localeCompare(b));
  const max = Math.max(...entries.map(([, n]) => n), 1);
  const barWidth = 100 / entries.length;

  return (
    <figure className="space-y-1">
      <svg
        viewBox={`0 0 100 ${height}`}
        preserveAspectRatio="none"
        className="w-full"
        style={{ height }}
        role="img"
        aria-label={`Probability distribution over ${entries.length} categories from ${samples} samples`}
      >
        {entries.map(([label, count], i) => {
          const h = (count / max) * (height - 24);
          return (
            <g key={label}>
              <rect
                x={i * barWidth + barWidth * 0.1}
                y={height - 18 - h}
                width={barWidth * 0.8}
                height={h}
                fill="var(--brand)"
              />
              <text
                x={i * barWidth + barWidth / 2}
                y={height - 4}
                textAnchor="middle"
                fontSize="6"
                fill="var(--muted-foreground)"
                className="font-mono"
              >
                {label}
              </text>
            </g>
          );
        })}
      </svg>
      <figcaption className="text-xs text-muted-foreground font-mono">
        {entries.length} categories · {samples} samples
      </figcaption>
    </figure>
  );
}
```

<Note id="types" kind="core" tip="Counts as Record<string, number> — the canonical TS shape for this primitive" remember="`Record<string, number>` matches Python's dict[str, int]. The interop type. Use this exact shape on every API boundary.">
Defining `Counts = Record<string, number>` once and exporting it makes the rest of the codebase consistent. Every API route that returns counts, every component that renders them, every test fixture — same type. This is what "designing for the data" looks like in TypeScript.
</Note>

<Note id="component" kind="core" tip="sort + max-derive at render time — no useState needed for a pure-derivation chart" remember="Pure render. No hooks. Sort + max derived from props. Re-renders cheap.">
This component is intentionally hookless — given the same `counts` prop, it renders identical output. No state, no useEffect, no memo. React's reconciliation handles re-render cost. The whole component is functional and trivially testable. Resist the temptation to add a `useMemo`; it costs more than it saves at this size.
</Note>

<Note id="svg" kind="mnemonic" tip="viewBox + preserveAspectRatio='none' = the chart scales to its container" remember="viewBox 0 0 100 H + preserveAspectRatio='none' = chart fills width responsively. No JS resize observer needed.">
The chart uses a 100-unit viewbox horizontally so each bar is a percentage. `preserveAspectRatio="none"` lets the SVG stretch to the container width without preserving aspect ratio of the bars — which is what you want for a histogram. No `useResizeObserver`, no layout-measuring effects.
</Note>

<Note id="legend" kind="gotcha" tip="Always render the sample count in the figcaption — never just bars" remember="figcaption with samples is non-negotiable. 4096 samples and 40 samples look identical without it.">
The chart visually looks the same whether you ran 40 samples or 4 million. The figcaption is the only thing telling the user how reliable the distribution is. This is the "don't hide the sample count" rule made concrete — it sits in the markup, not in a tooltip.
</Note>

```vue {1-13#script 15-37#template 39-43#styles} title="components/DistributionChart.vue"
<script setup lang="ts">
type Counts = Record<string, number>
const props = withDefaults(defineProps<{
  counts: Counts
  samples: number
  height?: number
}>(), { height: 160 })

import { computed } from 'vue'
const entries = computed(() => Object.entries(props.counts).sort(([a], [b]) => a.localeCompare(b)))
const max = computed(() => Math.max(...entries.value.map(([, n]) => n), 1))
const barWidth = computed(() => 100 / entries.value.length)
</script>

<template>
  <figure class="chart">
    <svg
      :viewBox="`0 0 100 ${height}`"
      preserveAspectRatio="none"
      :style="{ height: `${height}px` }"
      role="img"
      :aria-label="`Probability distribution over ${entries.length} categories from ${samples} samples`"
    >
      <g v-for="([label, count], i) in entries" :key="label">
        <rect
          :x="i * barWidth + barWidth * 0.1"
          :y="height - 18 - (count / max) * (height - 24)"
          :width="barWidth * 0.8"
          :height="(count / max) * (height - 24)"
          fill="var(--brand)"
        />
        <text :x="i * barWidth + barWidth / 2" :y="height - 4" text-anchor="middle" font-size="6" fill="var(--muted-foreground)">
          {{ label }}
        </text>
      </g>
    </svg>
    <figcaption>{{ entries.length }} categories · {{ samples }} samples</figcaption>
  </figure>
</template>

<style scoped>
.chart svg { width: 100%; }
figcaption { font-family: var(--font-mono); font-size: 0.75rem; color: var(--muted-foreground); margin-top: 0.25rem; }
</style>
```

<Note id="script" kind="cross" tip="computed() is Vue's equivalent of derived state at render time — same idea as Next's in-render derivation" remember="React: derive in render body. Vue: derive in computed(). Both run on every prop change. Pick the framework's idiom.">
Vue's `computed` is more explicit than React's "just do it in the function body." Both serve the same purpose: derive values from props/state with automatic re-evaluation. The Vue version caches by default (only recomputes if reactive deps change); React's in-body derivation re-runs every render. For tiny derivations like this, the difference is negligible.
</Note>

<Note id="template" kind="core" tip="v-for on <g> mirrors React's .map() — but the <g> doesn't need to be the keyed element" remember="v-for with :key on <g>. Vue auto-keys but explicit is better for reorders. Same data → same bars.">
Vue's `v-for` with `:key="label"` lets the framework efficiently reconcile when counts change — bars that share a label keep their DOM nodes, new bars get fresh nodes. This is the Vue version of React's `key={label}` on the array element. Without keys, Vue (and React) recreate all DOM on every prop change.
</Note>

<Note id="styles" kind="mnemonic" tip="scoped styles + global tokens — the design-tokens pattern in action" remember="Local styles for this component. var(--brand), var(--muted-foreground) from the token layer.">
This is where the `design-tokens-cross-stack` concept pays off. The bars are filled with `var(--brand)`; the figcaption uses `var(--muted-foreground)`. Switch the theme from light to dark — the chart adapts automatically without one line of component code changing. Tokens carry the design intent across components.
</Note>

<Note id="auto-import" kind="cross" tip="ref / computed / defineProps are all auto-imported in Nuxt — no import line" remember="Nuxt auto-imports all Vue reactivity primitives. Templates that look like magic; behavior is identical to explicit Vue.">
The Nuxt version omits the `import { computed } from 'vue'` line because Nuxt's auto-import system handles it at build time. The component is otherwise identical to the Vue version. Auto-imports save a few lines per file, but they hurt grep-ability — `which file imports computed` is harder to answer in a Nuxt codebase.
</Note>

<Note id="props" kind="core" tip="withDefaults + defineProps gives you typed props with default values — Vue's most modern API" remember="withDefaults(defineProps<{...}>(), { height: 160 }) — props type + defaults in one expression.">
This is the cleanest way to declare props with optional values and defaults in Vue 3. The TypeScript shape gets enforced at compile time; the defaults get inlined at runtime. Less ceremony than React's `Props = { height?: number }` + destructuring with `height = 160` separately.
</Note>

<Note id="template" kind="mnemonic" tip="Same template body as plain Vue — Nuxt doesn't change the component, just the imports" remember="Component identical to Vue tab. Nuxt adds auto-imports + auto-registration. The template logic is invariant.">
This is the cross-stack invariant made visible: take the Vue tab's template, drop it into the Nuxt tab, and it works without changes. Nuxt is "Vue with batteries" — the batteries (auto-import, file-based routing, SSR) don't change the component's body, just the surrounding scaffolding.
</Note>

<Note id="fetch" kind="gotcha" tip="useFetch with a typed generic — narrows the data shape so the chart's prop types are enforced" remember="useFetch<{ counts: ...; samples: ... }>('/api/distribution') → data.counts is typed end-to-end.">
The TypeScript generic on `useFetch` flows the API response shape through to the template. When you write `<DistributionChart :counts="data.counts" :samples="data.samples" />`, TypeScript checks that `data.counts` matches the `Counts` type the component expects. End-to-end type safety with zero ceremony — the kind of polish that makes a designer feel safe shipping fast.
</Note>
