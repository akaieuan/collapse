## The mental shift

Every quantum demo ships in two modes: **real hardware** (lab desk with a coupled rig) and **simulator** (user laptop). Maintaining two code paths is how teams end up with two divergent codebases. The clean solution is a single env flag (`USE_REAL_HARDWARE`) and an import-time fork that resolves to one of two backends — both implementing the same interface. The rest of the call site has no idea which it's talking to.

The contract:

- **One env var** — booleanish (`"true"` / unset = simulator) read once at module load.
- **Dual import** — `if real: from realdriver import X; else: from sim import X` (or framework-specific equivalents).
- **Identical signature** — `run(input) → result`, no branching at call sites.
- **Surface the mode** — log or render which backend is active so demos don't quietly lie.

```python {1-4#imports 6-8#flag 10-16#dual 18-24#api 26-31#main} title="device_controller.py"
import os
from typing import Protocol
from qiskit import QuantumCircuit
from qiskit_aer import AerSimulator

def read_mode_from_env() -> bool:
    return os.getenv("USE_REAL_HARDWARE", "false").lower() == "true"

USE_REAL = read_mode_from_env()

if USE_REAL:
    import clr  # pythonnet bridge to ThorLabs CLR driver
    from realdriver import PolarizerRig as _Backend
else:
    from fakedriver import IQMFakeAdonis as _Backend

class QuantumBackend(Protocol):
    def run(self, circuit: QuantumCircuit, shots: int) -> dict: ...

def control_real_hardware(input_circuit: QuantumCircuit, shots: int = 4096) -> dict:
    backend = _Backend()
    print(f"[mode] {'HARDWARE' if USE_REAL else 'SIMULATOR'}")
    return backend.run(input_circuit, shots=shots)

if __name__ == "__main__":
    qc = QuantumCircuit(1)
    qc.h(0)
    qc.measure_all()
    print(control_real_hardware(qc))
```

<Note id="imports" kind="core" tip="Import both backends behind the flag — never both at runtime" remember="Top-level import behind `if USE_REAL`. Heavy hardware deps (pythonnet, vendor SDKs) never load on user laptops.">
The whole point of the toggle is that the user-laptop path doesn't pay the import cost of vendor-specific drivers. If you import both eagerly, you've defeated the pattern — anyone running the simulator code on a machine without `pythonnet` crashes at import time.
</Note>

<Note id="flag" kind="mnemonic" tip="Read the env once, at module load — never re-read mid-run" remember="One read, one constant. Mid-run env mutations are not your friend.">
A function that re-reads env on every call invites subtle bugs (test sets env, forgets to unset, next test inherits). One module-level constant. If you need runtime switching, that's a different feature — usually means you need a registry pattern, not an env flag.
</Note>

<Note id="dual" kind="core" tip="Both backends implement the same Protocol — duck-typed, not nominally typed" remember="Protocol = structural typing. Both backends just need .run(circuit, shots) → dict.">
`Protocol` is the Python equivalent of a TypeScript interface — structural, not nominal. The two backends don't inherit from a shared class; they just need to expose the same method shape. This makes it trivial to add a third backend later.
</Note>

<Note id="api" kind="core" tip="Call sites never branch on USE_REAL — the toggle is invisible above this layer" remember="If `if USE_REAL` appears in business logic, the abstraction has leaked. Push it back down.">
The whole rest of the codebase calls `control_real_hardware(circuit)` and never knows whether the result came from silicon or simulation. If you see `if USE_REAL` outside this module, that's a code smell — push the branching back into the driver layer.
</Note>

<Note id="main" kind="gotcha" tip="Always surface which mode is active — silent mode-mismatch is the demo killer" remember="Log/print the mode at startup AND in every result envelope. Otherwise demos lie quietly.">
The most common bug: someone forgets to set the env var, demo runs on the simulator, results look "fine," team thinks the hardware works. Always log the mode at startup AND attach it to the response envelope, so debugging logs make it obvious.
</Note>

```ts {1-4#imports 6-9#flag 11-15#dynamic 17-31#post 33-35#response} title="app/api/run-circuit/route.ts"
import { NextResponse } from "next/server";
import { z } from "zod";

export const runtime = "nodejs";

const USE_REAL = process.env.USE_REAL_HARDWARE === "true";
const BridgeURL = process.env.BRIDGE_URL ?? "http://localhost:5000";

type RunFn = (input: unknown, shots: number) => Promise<unknown>;

const run: RunFn = USE_REAL
  ? async (input, shots) => fetch(`${BridgeURL}/api/run`, { method: "POST", body: JSON.stringify({ input, shots }) }).then((r) => r.json())
  : async (input, shots) => ({ stub: true, input, shots, note: "simulator stub — wire to AerSimulator via Python sidecar for real values" });

const RequestSchema = z.object({
  circuitJson: z.unknown(),
  shots: z.number().int().positive().max(100_000).default(4096),
});

export async function POST(req: Request) {
  const parsed = RequestSchema.safeParse(await req.json().catch(() => null));
  if (!parsed.success) {
    return NextResponse.json({ error: "invalid_request" }, { status: 400 });
  }
  const result = await run(parsed.data.circuitJson, parsed.data.shots);
  return NextResponse.json({
    mode: USE_REAL ? "hardware" : "simulator",
    result,
  });
}
```

<Note id="imports" kind="mistake" tip="runtime: 'nodejs' — Edge has no node-internal fetch privileges for the bridge" remember="Pin nodejs runtime. Edge will tunnel to bridge fine, but you lose env access patterns.">
Edge runtime can't read `process.env.USE_REAL_HARDWARE` the same way Node can — env access is whitelisted via `NEXT_PUBLIC_` prefixes only. Pinning Node makes the env-flag pattern work without rewiring.
</Note>

<Note id="flag" kind="core" tip="One read, at module load — same as the Python pattern" remember="`process.env.USE_REAL_HARDWARE === 'true'` once. Don't read mid-handler.">
Same rule as Python: read once at module load. If you read inside the handler, every test that mutates env affects subsequent requests in the same warm Lambda.
</Note>

<Note id="dynamic" kind="core" tip="Branch at module load — runtime conditional means both paths are bundled" remember="`const run = USE_REAL ? ... : ...` — assigns the right implementation once. Use dynamic import() if hardware code has heavy deps.">
For lighter cases, ternary at module scope is fine. If the hardware path imports a heavy SDK (vendor DLL bindings, vendor-specific WebUSB), use `dynamic import()` inside the ternary so the simulator path never loads the SDK at all.
</Note>

<Note id="post" kind="gotcha" tip="Validate before dispatching — Python and hardware are equally bad places for malformed input" remember="Zod parses BEFORE the run() call. Hardware errors are 10× more expensive than Zod errors.">
Hardware time is precious. Don't let a typo'd `shots: -1` consume a real quantum job slot. Validate at the Node boundary, surface errors as 400s, save the hardware path for valid inputs.
</Note>

<Note id="response" kind="mnemonic" tip="Always attach the mode to the response — make it grep-able in client logs" remember="`{ mode: 'hardware' | 'simulator', result }` — never just `result`. The mode survives all downstream logging.">
Mirrors the Python pattern of logging the mode. The client can show a "running on simulator" badge, telemetry can split metrics by mode, support can grep logs by mode. Costs you one extra field; saves you a debugging session.
</Note>

```vue {1-3#config 5-13#composable 16-22#template} title="composables/useQuantumRun.ts + RunPanel.vue"
<script setup lang="ts">
import { ref } from 'vue'
import { useRuntimeConfig } from '#imports' // (or your equivalent injection in plain Vue)

function useQuantumRun() {
  const config = useRuntimeConfig().public
  const mode = config.useRealHardware ? 'hardware' : 'simulator'
  const result = ref<unknown>(null)
  const pending = ref(false)
  async function run(circuitJson: unknown, shots = 4096) {
    pending.value = true
    const res = await fetch('/api/run-circuit', { method: 'POST', body: JSON.stringify({ circuitJson, shots }) })
    result.value = await res.json()
    pending.value = false
  }
  return { mode, result, pending, run }
}
</script>

<template>
  <div>
    <span class="badge">mode: {{ mode }}</span>
    <button @click="run(currentCircuit)" :disabled="pending">{{ pending ? 'Running…' : 'Run' }}</button>
    <pre v-if="result">{{ result }}</pre>
  </div>
</template>
```

<Note id="config" kind="cross" tip="Vue's runtimeConfig is the framework-agnostic equivalent of process.env" remember="In plain Vue: inject via provide/inject or import.meta.env. In Nuxt: useRuntimeConfig().public.">
Plain Vue gets env via `import.meta.env.VITE_USE_REAL_HARDWARE`. Nuxt gets it via `useRuntimeConfig().public.useRealHardware`. The shape differs; the pattern doesn't.
</Note>

<Note id="composable" kind="core" tip="Mode is determined once per composable instance — surfaced as a returned ref" remember="`mode` is in the return object. Templates use it for badges; logic uses it for branching only if absolutely necessary.">
The mode is computed at composable setup, not re-evaluated on every call. The component can render a "Hardware" / "Simulator" badge from `mode` without re-reading config. Same one-read principle as Python and Next.
</Note>

<Note id="template" kind="mnemonic" tip="Show the mode in the UI — never let users guess which they're running" remember="`{{ mode }}` in the template. Demo accountability: the user always knows what's behind the button.">
A small "mode: simulator" or "mode: hardware" badge near the run button. Costs nothing visually; catches the "I thought I was on hardware" demo bug every time.
</Note>

```ts {1-7#runtime-config 10-20#server-handler 22-29#client} title="nuxt.config.ts + server/api/run-circuit.post.ts + pages/run.vue"
// nuxt.config.ts
export default defineNuxtConfig({
  runtimeConfig: {
    public: {
      useRealHardware: process.env.USE_REAL_HARDWARE === 'true',
    },
    bridgeUrl: process.env.BRIDGE_URL ?? 'http://localhost:5000',
  },
})

// server/api/run-circuit.post.ts
export default defineEventHandler(async (event) => {
  const cfg = useRuntimeConfig()
  const body = await readBody(event)
  const result = cfg.public.useRealHardware
    ? await $fetch(`${cfg.bridgeUrl}/api/run`, { method: 'POST', body })
    : { stub: true, ...body }
  return { mode: cfg.public.useRealHardware ? 'hardware' : 'simulator', result }
})

// pages/run.vue (client)
<script setup lang="ts">
const cfg = useRuntimeConfig().public
const { data, pending, error, execute } = await useFetch('/api/run-circuit', {
  method: 'POST',
  body: { circuitJson: getCircuit(), shots: 4096 },
  immediate: false,
})
</script>
```

<Note id="runtime-config" kind="core" tip="public keys are visible client-side; private keys are server-only — keep the bridge URL private" remember="`runtimeConfig.public.X` = client-visible. `runtimeConfig.X` = server-only. Bridge URL = server-only. Mode flag = public (for the badge).">
Nuxt's two-tier runtimeConfig is the cleanest API of any framework here. Mode goes public (the client needs to render the badge). The bridge URL stays private (no need to expose your ngrok tunnel to the browser bundle).
</Note>

<Note id="server-handler" kind="cross" tip="Same toggle, same shape, less ceremony than Next" remember="$fetch + defineEventHandler = Next route handler. No NextResponse wrapping. Return the body, status defaults to 200.">
Compare to Next's `app/api/run-circuit/route.ts`: same logic, fewer lines, no `NextResponse.json()` wrapping. Nuxt's Nitro is the Node story without the React-framework ceremony.
</Note>

<Note id="client" kind="mnemonic" tip="immediate: false delays the call until execute() is clicked — useful for hardware time" remember="useFetch({ immediate: false }) — no auto-run. Pair with a button click. Hardware time isn't free.">
On hardware mode, you don't want a circuit to fire the moment the page loads. `immediate: false` defers the call; the user clicks to start. On simulator mode, you can leave it eager since simulator runs are nearly free.
</Note>
