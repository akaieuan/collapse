## The mental shift

You have a heavy library you want to use (a native addon, a full SDK, a WASM blob) and a lighter alternative that covers 80% of the API surface (a pure-JS fallback, a stub, a JSON shape). Instead of writing two versions of every function, you write **one** — but the import line and a sentinel flag let it adapt. When the heavy library loads, the full code path runs. When it doesn't, the lighter library takes over and a flag (`isLight = true`) disables the paths the lighter version can't support.

The contract:

- **Try heavy first** — `try { await import("sharp") }`. If it loads, great.
- **Catch → fall back** — `catch { await import("./light-blur") }`. Same import names, same surface API.
- **Sentinel flag** — `isLight = true` tells the rest of the file what's loaded.
- **Gate advanced paths** — `if (isLight) return simpleResult` short-circuits anything the lighter library can't do.
- **Surface the mode** — never silently degrade. Tell the user (or at least the logs) which library won.

```ts {1-3#types 5-21#dynamic 23-32#handler 34-42#feature} title="lib/blur.ts + app/api/run/route.ts"
// lib/blur.ts
type BlurImpl = {
  blurImage: (px: number[], radius: number) => Promise<number[]>;
  isLight: boolean;
};

let _impl: Promise<BlurImpl> | null = null;
function loadBlur(): Promise<BlurImpl> {
  if (_impl) return _impl;
  _impl = (async (): Promise<BlurImpl> => {
    try {
      const heavy = await import("sharp"); // native image library (may be unavailable)
      return { blurImage: heavy.blurImage, isLight: false };
    } catch {
      const light = await import("./light-blur");
      return { blurImage: light.blurImage, isLight: true };
    }
  })();
  return _impl;
}

// app/api/run/route.ts
import { NextResponse } from "next/server";
export const runtime = "nodejs";

export async function POST(req: Request) {
  const { pixels, radius } = await req.json();
  const impl = await loadBlur();
  const result = await impl.blurImage(pixels, radius);
  return NextResponse.json({ mode: impl.isLight ? "light" : "heavy", result });
}

// Components can also check the mode (server component or via API)
async function getMode() {
  const impl = await loadBlur();
  return impl.isLight ? "Lightweight (pure-JS)" : "Native (sharp)";
}
```

<Note id="types" kind="core" tip="Type the unified interface FIRST — both implementations must satisfy it" remember="`BlurImpl` defines the shape. Both branches return a value that conforms. The rest of the app uses the type, not the source.">
TypeScript's structural typing makes this cleaner than Python's Protocol. Define the interface once; both implementations satisfy it; consumers depend only on the interface. The bundler still tree-shakes whichever side wins.
</Note>

<Note id="dynamic" kind="core" tip="Memoize the impl promise — every call resolves the same instance" remember="`let _impl: Promise<...> | null = null` at module scope. Only the first call does the import dance.">
Without memoization, every request to `/api/run` re-tries the heavy import, which is slow (the failure path includes the full dynamic-import resolution). Cache the resolved promise; subsequent calls await the cached value. Memory cost: one promise.
</Note>

<Note id="handler" kind="cross" tip="Surface the mode in the response — same as the Python pattern's print()" remember="`{ mode: 'light' | 'heavy', result }`. Clients render a badge or split telemetry.">
This is the JS equivalent of the Python `print(f"[mode] ...")`. The mode rides along on every response so the client (or your logs) can show what's actually running. Demo-day insurance.
</Note>

<Note id="feature" kind="gotcha" tip="Don't try-catch around dynamic import() at every call site — centralize the fallback" remember="One place handles the try/catch (lib/blur.ts). API routes/components just await loadBlur().">
Bad pattern: every API route does its own `try { import heavy } catch { import light }`. You'll end up with subtly different fallback behavior in different routes. Centralize the resolver in `lib/blur.ts` (or wherever) and depend on the unified `BlurImpl` everywhere.
</Note>

```vue {1-3#composable 5-19#resolve 22-31#use 34-40#template} title="composables/useBlur.ts + BlurDemo.vue"
<script setup lang="ts">
import { ref, computed, onMounted } from 'vue'

type Impl = { blurImage: (px: number[]) => Promise<number[]>; isLight: boolean }

const impl = ref<Impl | null>(null)
const mode = computed(() => {
  if (!impl.value) return 'loading'
  return impl.value.isLight ? 'light' : 'heavy'
})

onMounted(async () => {
  try {
    const heavy = await import('sharp')
    impl.value = { blurImage: heavy.blurImage, isLight: false }
  } catch {
    const light = await import('./light-blur')
    impl.value = { blurImage: light.blurImage, isLight: true }
  }
})

const result = ref<number[] | null>(null)
const pending = ref(false)
async function run(pixels: number[]) {
  if (!impl.value) return
  pending.value = true
  result.value = await impl.value.blurImage(pixels)
  pending.value = false
}
</script>

<template>
  <div>
    <span class="badge" :data-mode="mode">{{ mode }}</span>
    <button @click="run(currentPixels)" :disabled="pending || !impl">
      {{ pending ? 'Blurring…' : mode === 'loading' ? 'Loading library…' : 'Blur' }}
    </button>
    <pre v-if="result">{{ result }}</pre>
  </div>
</template>
```

<Note id="composable" kind="cross" tip="Same fallback shape as the Next route — just in a composable instead of a module" remember="Try heavy → except → light. The composable exposes mode + run().">
Vue's composable replaces the Node module-level resolver. The pattern is identical: try to load the heavy library, fall back to the light one, expose a ref for the resolved implementation. Each component that uses the composable gets its own instance, but onMounted ensures the import only runs once per mount.
</Note>

<Note id="resolve" kind="mistake" tip="onMounted, not setup — dynamic import in setup() is SSR-incompatible" remember="`onMounted(async () => ...)` — only run on the client. Dynamic imports of native deps in setup() will crash SSR.">
If the heavy library has any browser-specific deps (Web Audio, WebGPU, DOM), importing it in `setup()` will execute during server-side rendering and crash. `onMounted` guarantees client-only execution. Trade-off: there's a brief "loading library…" state on first mount.
</Note>

<Note id="use" kind="gotcha" tip="Guard the run() call — impl might not be resolved yet" remember="`if (!impl.value) return` at the top of run(). The button is disabled until resolved.">
Race condition guard: the user clicks "Blur" before the dynamic import finishes. Without the guard, you'd hit `Cannot read property blurImage of null`. With it, the button is disabled and the click is a no-op until the library lands.
</Note>

<Note id="template" kind="mnemonic" tip="Render the mode as a data attribute — CSS-style it without leaking config into props" remember="`:data-mode='mode'` — three states: 'loading', 'light', 'heavy'. Color-code with CSS attribute selectors.">
`[data-mode="light"]` and `[data-mode="heavy"]` can be styled differently in CSS — yellow badge for light mode, green for heavy, gray pulse for loading. UX of "which library is in charge" is now visible and styleable without prop drilling.
</Note>

```ts {1-9#plugin 11-23#composable 26-32#page} title="plugins/blur.client.ts + composables/useBlur.ts + pages/blur.vue"
// plugins/blur.client.ts  (only runs on the client — note the .client suffix)
export default defineNuxtPlugin(async () => {
  try {
    const heavy = await import('sharp')
    return { provide: { blur: { ...heavy, isLight: false } } }
  } catch {
    const light = await import('./light-blur')
    return { provide: { blur: { ...light, isLight: true } } }
  }
})

// composables/useBlur.ts
export function useBlur() {
  const { $blur } = useNuxtApp()
  const mode = $blur.isLight ? 'light' : 'heavy'
  const result = ref<number[] | null>(null)
  const pending = ref(false)
  async function run(pixels: number[]) {
    pending.value = true
    result.value = await $blur.blurImage(pixels)
    pending.value = false
  }
  return { mode, result, pending, run }
}

// pages/blur.vue
<script setup lang="ts">
const { mode, result, pending, run } = useBlur()
</script>

<template>
  <div>
    <span class="badge">mode: {{ mode }}</span>
    <button @click="run(getPixels())" :disabled="pending">{{ pending ? 'Blurring…' : 'Blur' }}</button>
  </div>
</template>
```

<Note id="plugin" kind="core" tip="The .client.ts suffix tells Nuxt this plugin runs ONLY on the client — bypasses SSR" remember="`plugins/blur.client.ts` — auto-registered, client-only. SSR ships placeholder; client hydrates with real impl.">
Nuxt's file-based plugin convention solves the SSR problem cleanly. `.client.ts` plugins never run on the server. SSR renders with `$blur` undefined; the client plugin runs on mount, hydrates `$blur`, and the app becomes interactive. The mode badge can show "loading" during this window.
</Note>

<Note id="composable" kind="cross" tip="$blur is the auto-injected app helper — Nuxt's DI without React's context boilerplate" remember="`useNuxtApp().$blur` accesses anything provided by a plugin. Composables wrap it for typed ergonomics.">
Compare to Next: you'd need a React Context, a Provider in the layout, a useContext hook, and a TypeScript declaration file. Nuxt does it with one plugin file and a typed `useNuxtApp()` call. The provide/inject pattern is the killer feature for cross-cutting concerns like this.
</Note>

<Note id="page" kind="mnemonic" tip="Pages stay clean — all the fallback complexity lives in the plugin layer" remember="The page only sees `mode`, `result`, `pending`, `run`. The try/except is buried under three layers of abstraction.">
This is the well-architected version: the page has zero knowledge of the fallback mechanic. If you swap the pure-JS fallback for some future even-lighter alternative, you change one file (the plugin). Pages, components, and other composables don't notice.
</Note>
